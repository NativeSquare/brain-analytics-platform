# Epic 7 Retrospective: Design System Alignment

**Date:** 2026-04-01
**Epic:** 7 -- Design System Alignment
**Stories:** 4/4 completed (100%)
**Total Points:** ~19 (3 + estimated 5 + estimated 3 + 8)
**First retrospective:** Yes (no previous retro found)

---

## 1. What Went Well

- **Clean execution across all 4 stories.** Three out of four stories (7.1, 7.2, 7.3) passed review with zero issues. This is exceptional for a porting epic where subtle visual regressions and API mismatches are common.
- **Spec quality was high.** Story specs were detailed with BDD acceptance criteria, exact CSS values, SVG coordinates, and component prop signatures. This left little room for ambiguity, which contributed to the clean reviews.
- **Story 7.4 lint fix was caught and resolved efficiently.** The React compiler error ("Cannot create components during render") from dynamically resolved Lucide icons was identified in review and fixed with `createElement` -- a clean, idiomatic solution.
- **Convex backend tables delivered.** `userPinnedDashboards` and `userRecentDashboards` tables with indexes were created correctly, providing the real-time subscription foundation for Epic 9.
- **Vendored dependency (simpleheat).** The canvas heatmap library was vendored as TypeScript at `apps/web/src/lib/simpleheat.ts`, avoiding an npm dependency on an unmaintained library.

## 2. What Could Be Improved

- **Story statuses not updated consistently.** Story 7.1 and 7.4 still show `status: draft`, Story 7.3 shows `ready-for-dev`, while only 7.2 shows `done`. Sprint-status.yaml does not exist at all. This makes automated tracking impossible.
- **No automated tests delivered.** None of the 4 stories included unit or integration tests for the ported components. For visual/SVG components this is understandable, but the Convex mutations (pin/recent tracking) and coordinate mapping logic (StatsBomb transform) are testable and should have tests before downstream epics depend on them.
- **No dev notes or implementation logs in story files.** The story specs contain only the original requirements -- no post-implementation notes, challenges encountered, or lessons learned sections were added. This reduces retrospective depth.

## 3. Risks and Concerns for Epics 8-11

- **Epic 9 directly depends on Story 7.4** (DashboardCardItem, gallery grid, pin/recent tables) and **Story 7.2** (pitch components for Shot Map and Heat Maps dashboards). These are untested.
- **Epic 10 depends on Story 7.3** (XYScatterChart for Player Analysis scatter plot, FilterBar/FilterSelect for all dashboards). These components live in `apps/admin/` -- see point 5 below.
- **No integration validation.** The ported components have not been tested with real data from Epic 8's API routes. First real integration will happen in Epic 9, where issues may surface.

## 4. Spec vs. Implementation Gaps

- **No significant gaps detected.** All acceptance criteria artifacts are present:
  - globals.css has OKLCH palette, Avenir Next font stack, 0.625rem radius (7.1)
  - All 4 pitch components + simpleheat + types file exist (7.2)
  - XYScatterChart, FilterBar, FilterSelect, FilterCheckbox, StatsItem all exist with index barrel files (7.3)
  - DashboardCardItem, DashboardGalleryGrid, dashboard-icons, Convex tables all exist (7.4)
- **Minor:** Story 7.4 spec mentions `allowedRoles` rendered as individual `Badge` components. Implementation shows a count badge ("3 roles") in the header plus individual role badges in CardContent. This is arguably better UX but diverges slightly from spec wording.

## 5. Component Placement Consistency

**This is a concern.** The split is:

| Location | Components |
|----------|-----------|
| `apps/web/src/components/dashboards/` | PitchBase, FullPitchBase, GoalBase, HeatPitchMap (Story 7.2) |
| `apps/web/src/components/dashboard/` | DashboardCardItem, DashboardGalleryGrid (Story 7.4) |
| `apps/admin/src/components/charts/` | XYScatterChart (Story 7.3) |
| `apps/admin/src/components/dashboard/` | FilterBar, FilterSelect, FilterCheckbox, StatsItem (Story 7.3) |

**Issues:**
1. **Chart/filter components in `apps/admin/` but pitch/card components in `apps/web/`.** Epic 9 and 10 dashboards will need both sets. If dashboards are built in `apps/web/`, the admin-side chart components will need to be either duplicated or moved to a shared package.
2. **Naming collision:** `dashboards/` (pitch components) vs `dashboard/` (card components) in `apps/web/` is confusing and will cause import mistakes.

**Recommendation:** Before starting Epic 9, evaluate whether chart components should be moved to `packages/ui/` or `apps/web/` depending on where dashboards will be rendered.
